The secret to great Git commit messages

3/6/2015

0 Comments

 
How to use Git in such a way that will make your friends and co-workers not want to kill you.
I've been using Git for a number of years, between personal projects at home, and collaborative projects at work.
Many of the projects are maintained for many years, and sometimes managed by many different people.

When sorting through a project, I can guarantee that there will come moments when you read some code and mutter under your breath "Why the heck is that done that way?".
At that very second, your first instinct should be to look at the history, to see who did what, trace it to a Commit Message, and figure out why.
If you don't already know how to to that, take moment to figure out a way quickly to access the history of a line in your editor or in your repo. The less friction it causes, the easier it is to go back and read about the changes. 
If you're looking for suggestions, how about Vim-Fugitive? A second terminal with git-blame?
If you make a habit out of always checking the git history, you will quickly learn what type of Git Commit message is helpful and what is not. 

Then, if you are like me, you'll want to go back to slap your past-self in the head for being lazy and giving a few bullet points of "what" was changed and not "why" it was changed.
In my experience, the best way to ensure that you have a reasonable commit message is to always include the word 'because'.
Let's take a peek at an example, to illustrate the power of 'because'.

First up, here is a typical commit message that I'm sure pretty much everyone has seen or dealt with at some point:
Changed the post title font size 

Changed 0.8em 16px. 

h2 { 
 - font-size: 0.8em;
+ font-size: 16px; 
} 
Unfortunately, this commit message gives absolutely zero context as to why it was changed.
Did we just not like that size? Was there a vendetta against em scaling of fonts? Is someone just trying to screw with me?
Yes, of course we know that the size was changed from 0.8em to 16px: the git diff is included right there! Why tell me twice?

Without any further context, it's almost impossible to discern the reasoning behind why this single line of code was changed.
Now, let's examine a 'proper' commit message:
Changed the post title font size 

Change H2 font to 16 pixels because when the overall font size was increased, the blog titles were wrapping on mobile views. 

h2 { 
- font-size: 0.8em; 
+ font-size: 16px; 
} 
You'll notice right away that the message includes a 'why?' solution. I know right away what it does, how it works, and what was broken.

However, most importantly, you'll also notice that it doesn't reiterate the same information included in the diff.
This second changelog message is most helpful.
Here's why:
  • It provides context to the change, rather than regurgitating the same info for the sake of including a message. 
  • It provides information as to the reasoning behind it and what might break if it is reverted. 
  • It's completely and totally obvious that it isn't some arbitrary change. 

I will appreciate all of these facts in 2 years when I'm looking back at it.
I'd like to see anyone else find a better way to communicate with their future selves that doesn't involve time travel.

How to install "Packer", the ArchLinux AUR Helper

6/23/2013

8 Comments

 
Packer is an extremely useful 'wrapper' for ArchLinux's Arch User Repository (AUR). According to the ArchWiki:
  • packer — wrapper for pacman and the AUR. It was designed to be a simple and very fast replacement for the basic functionality of Yaourt. It has commands to install, update, search, and show information for any package in the main repositories and in the AUR. Use pacman for other commands, such as removing a package
Every time I install Arch, I find myself trying to remember how I installed packer. Many of the times, I'm trying to do this via command line only access. 

To install Packer:

I find it easiest to do everything quickly from the command line.
To install Packer:
A quick note regarding the commands:
Lines that are prefaced with a 'hash' symbol (#) are to be run as root user
Lines that are prefaced with a 'dollar' sign ($) are to be run as a regular user
  1. Install the package dependencies needed to use and build Packer:
    #: pacman -S base-devel fakeroot jshon expac git
    (You would also need bash, curl, pacman, grep, and sed but they're typically part of the base install)
  2. Get the PKGBUILD (I typically save it in a separate folder for cleanliness)
    #: wget https://aur.archlinux.org/packages/pa/packer/PKGBUILD
  3. Run makepkg on the PKGBUILD as a regular user
    $: makepkg
  4. Install the newly created package
    #: pacman -U packer-*.pkg.tar.gz

To Use Packer:

Packer works exactly the same as Pacman.

As a regular user you can run:
$: sudo packer -S newpackage
to install the package you wish, or you can run 
$: sudo packer -Ss something
to find a package based on your 'something' search term.

However, remember that Packer will not let you remove a package. Use regular pacman to remove an unneeded package from your system.

Penn Manor - 1 Year In

12/13/2012

0 Comments

 
Just a few weeks ago, I passed my 1 year 'anniversary' at Penn Manor.
I'd like to take this opportunity to reflect on two of the awesome projects that I've been able to do over the past 12 months. 

Student Linux Machines

The biggest project of which I've been a part would be the creation of "Comet Linux". 
This is a totally customized version of Ubuntu Linux, created specifically for the students in grades 1-12. 
Screenshot of an early version of Comet Linux
We've worked very hard to include features based upon feedback from staff. 

The OS uses XFCE as the window manager, and at it's core is Ubuntu 12.04 LTS. 
The machines are managed through the use of a program called Puppet. 
Puppet allows us to explicitly define files and folders on the system, creating the perfect method for setting up a stable student system. 
Currently, the code for Comet Linux is closed source, but we hope to make it available to others in the future. 
The most interesting feature about our OS setup is the fact that the computers have the ability to 'self heal'. This means that if a student accidentally changes settings or deletes a needed file, one would simply have to log out, and then log back in to fix it. 
The profiles are stored with a 'skeleton' setup that is copied over any changes made whenever a user logs in. 
Students save their files on a remote server, not only to back them up, but to access them from anywhere in the school district.
Because all 4 of our Tech Support Analysts work on the maintenance of the Linux machines, we need a way to manage the updates. To make things easier, we have a Git server set up.
Git is a version control program that allows multiple people to work on the same files at the same time, all while avoiding the inevitable 'too many cooks in the kitchen' problem. 
All in all, the Comet Linux OS is one of my favorite projects to have worked. 

Computer Builds

Very recently, the entire Tech Team and I have worked to build 40 computers for custodians, nurses, and cafeteria workers throughout the district. 
We chose to build the machines ourselves to save quite a bit of money, as compared to purchasing a pre-built system. 
We created a 'Master' computer, configured to include everything that the staff would need, and created an image using the utility FOG. FOG is a program like Clonezilla (an imaging program that I've discussed earlier) but it operates at high speeds, making it ideal for a project such as this.
There are a number of pictures available from this event also available on the Penn Manor Techblog.

Raspberry Pi - A Low Power Solution

12/13/2012

1 Comment

 
The Raspberry Pi is a credit-card-sized single-board computer developed in the UK by the Raspberry Pi Foundation with the intention of stimulating the teaching of basic computer science in schools.
I was very interested in the Raspberry Pi from the start, as it is a very capable and modestly priced device. I'd like to discuss the various tasks that I have my  personal Raspberry Pi perform.

About My Pi

My Raspberry Pi is one of the first generation Model B devices. 
This Pi has 128Mb of RAM, and I have a 32Gb SD Card installed. 

I have used the ArchLinux ARM OS on the device, as I do not have any need for a graphical user interface.

Remote Access

Remote access for any Unix user is old hat; simply install SSH on the device and configure it for your needs.
For Arch Linux, it's as easy as running the command:
packer -S ssh
and assuring that the passwords for your users are correct.
However, I went one step further and configured my Pi as the single point of entry for my home network. 
I have several devices on my network at any time; a Pogoplug, a file server, and a full desktop tower that is configured for XBMC. Rather than configure my network to open access for all of these devices, I simply pointed them all to the Pi, and configured the Pi as a 'gatekeeper'.
If I ever want to access my internal network while on the go, I simply connect to my Pi, then initiate a connection to the device in question. A simple 2-step process with SSH. 

File Storage

Before I had my Raspberry Pi, I had my main desktop computer running all of the time, so that I could access movies, music, and other files when I wanted. However, the desktop computer uses quite a bit of power to keep running at all times. 
I decided that it would be most economical to swap all of my file storage needs to my Pi. 
Once I plugged my external hard drives into the Raspberry Pi, I configured SAMBA, a file server program, according to the ArchLinux Wiki. After this was all configured, I could easily access my files from other computers, even my tablets and phones!

XBox 360 Streaming

One feature that I was anxious to try was XBox 360 streaming. 
Typically, users could set up their Windows computer to share music and movies with their XBox 360.
The best program that I've found for streaming to an XBox 360 on Linux is MiniDLNA. 
Again, following the setup instructions on the ArchLinux Wiki, I was able to quickly configure my Raspberry Pi to share music and movies. In no time at all, I was able to listen to my music while playing a game. 

Others?

I'm still finding ways for my Raspberry Pi to replace functions that my larger, and more power-hungry computers would do. 

As I find them, I'll keep you posted!
Have you found any useful ways that your Raspberry Pi has come in handy? 
Let me know in the comments!

Bulk Hard Drive Imaging with CloneZilla and Samba

10/20/2011

5 Comments

 
It seems that this job at the Mahanoy Area School District is giving me quite a few things to talk about! I'm certainly not complaining. 
One interesting task that I had this past month or so was to upgrade multiple laptop carts that teachers use to aid their lessons. 

CLONEZILLA
Using my previous Open Source experience, I knew about a specialized Linux distribution called CloneZilla. 

I have used this program in the past when I planned to upgrade hardware on my computers at home, and was too afraid that I would cause data loss. (What can I say? I was cautious with my animated .gifs!)

Using the knowledge I gained from tinkering with it in the past, I was able to quickly and easily jump in and use the program to create images and restore the images from an attached hard drive. 
The problem that arose was this; it takes about 1 hour for the program to run, and until it's done, your only external hard drive is occupied by one greedy little program. 

This becomes a massive problem when you have 6 carts full of computers, nearly 180 laptops in total. Rather than spending 180 hours doing each laptop individually, I knew there had to be a better way. 

Networks to the rescue!

Source: http://frrl.wordpress.com/2009/02/15/let-the-republic-of-china-backup-your-linux-and-windows-pc/
I began to play around with the other settings of CloneZilla. In the image above, you will see the various options that you may use. 

Up until this time, I only had experience using 'local dev'; local devices such as hard drives plugged in with the backup images. 

You will also notice an option for SAMBA. Samba is the Linux name for Windows Shared Folders. 
This had me thinking: I knew that our office had about 6 Windows servers, and they all had shared folders available. I began to tinker again. 

We upgraded a single laptop and installed all of the necessary programs, software, and passwords, and then took an image. I found that you could back images up to the Samba servers, and then restore them over the network. I tried it with a single laptop in my office and it seemed to work well enough, so I began to scale the project up. 
My supervisor and I took a laptop cart into an unused classroom and began to restore the images from the server.
CloneZilla does an excellent job of walking you through the setup process. Once we specified that we wanted a Samba restore, you simply enter the fully qualified domain name of the server, the domain name of your network, a username that has access to the folder, and your password. 
Switches and Wires, Gigabit and Cat5!
The first few runs worked very well with just a few laptops running at a time. But we decided to scale the project even further. 
I noticed that on the boot menu for CloneZilla, there is an option to Boot to RAM. 
Source: http://www.techmixer.com/clonezilla-live-cd-clone-your-disk-and-disk-partition-for-free/
So, we decided that we should see how many laptops we could image at one time. 
For a while, we were doing 15 at a time, booting to RAM and then pulling the image from the Samba server. 
Naturally, I wanted to push the limit and see how many laptops we could image at one time. 
Well, today I got my answer. 
I did a whole cart at once. 
30 laptops all running from RAM connected to a central switch and pulling an image from a private hosted Windows Share folder. 
As one would assume, the network bottleneck was pretty atrocious. We were able to run the 80 GB image to all 30 of the laptops in just a little over 2.5 hours. 

In closing,

CloneZilla is an amazing program, and is an invaluable time saving tool. Using this program, we were able to take a project that should have taken 60 hours at the least and blast through in just under a single work week.

nComputing and You

10/12/2011

10 Comments

 
When I first started working at Mahanoy Area School District this past May, I started to investigate many of the interesting technologies that they had implemented around the school. I had noticed that there were quite a few nComputing devices being used throughout the school, but I had no idea what they were. Little did I know that they are probably the coolest little devices I have ever seen. 

nComputing vSpace 

Source: http://goo.gl/gtkwx
Seen at left is the complete nComputing device. The nComputing System is a small scale virtual computing device. It allows users to expand computer resources for a very low cost. This is what originally drew my supervisor into using the devices within the school. 
Source: http://goo.gl/BK6zn
These diminutive device attach to the back of a computer monitor, and are powered by USB. 
In the image, as seen from  left to right the ports available are; VGA out to a monitor, microphone in, audio out, USB tether for power, and 2 USB ports for connecting a mouse and keyboard. 
These devices must all be attached to a 'host' computer, running Windows or Linux. The host computer runs some very lightweight software that allows the connected devices to communicate, essentially making the nComputing devices a 'remote desktop' client. 
So for the cost of 1 full fledged desktop, and several monitors, you can deploy a fleet of computers quickly and easily. 

The maintenance on the devices are almost nonexistent. Since I have been working with them, the only problems that we have encountered were teachers not knowing that the 'host' computer must be turned on for the other devices to work. 
The nComputing devices come in 3 model types. The L-Series (using Ethernet), the U-Series (using USB), and the X-Series (using PCI cards). 
After experiencing the L-Series and the U-Series, I prefer the newer U-Series models better. 

Mahanoy Area's setup uses the L-Series in several of the larger libraries because of the space requirements. These devices only need the monitor, keyboard and mouse, and the nComputing device plugged and 'piggybacking' on the monitor. After plugging the monitor in, the only other connection is a single Ethernet connection that pulls information from a central server (in our case, a Windows Server 2003 system). These work very well in the circumstances, as they are only used every few days for about an hour at a time. The current network infrastructure isn't strong enough to support much more than that. 

The U-Series devices are my favorites. As mentioned before, they are USB powered, and run off of a local host. We were able to set up a computer lab for 30 students using only 6 full-fledged desktops because of this feature. So long as they are either plugged directly into a PCI USB card on the main computer, or into a powered USB hub, they run exceptionally well. When it is necessary to update the operating system (typically on Microsoft's 'Patch Tuesday') you only need to assure that the updates are applied to the host computers, and the updates will be passed along to the nComputing clients. This significantly cuts down time spent managing the computers. 
All in all, I would definitely recommend these devices to those who are looking to get a lot of computers quickly and on the cheap.
